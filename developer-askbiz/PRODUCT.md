# Product

## Register

product

## Users

Two overlapping groups sharing the same dashboard: professional developers integrating AskBiz's business-AI API (vision recognition, WhatsApp messaging, natural-language BI, merchant billing) into their own software, and non-technical small business owners self-serving the same surfaces with little to no coding background. Context: evaluating on a laptop or phone whether AskBiz is worth building on, or mid-integration checking usage, keys, and billing. Africa-wide SMB market — English is often not a first language, and reading long technical prose is a real friction point, not an edge case.

## Product Purpose

developer.askbiz.co is the API platform for AskBiz: key management, billing, webhooks, a live console, and public docs. Success is a developer or non-technical owner going from signup to a working, safe first API call in minutes, and trusting the platform enough to eventually put real money and real customer messages through it.

## Brand Personality

Quietly capable, plain-spoken, never condescending. Competent-utility over flashy-startup theater. Grounded in the real African SMB context this platform serves, not Silicon-Valley SaaS bravado. Plain language should work for a senior engineer and a first-time shop owner without insulting either.

## Anti-references

Not generic-SaaS-cream gradient-hero dashboard tropes. Not enterprise-cold, over-corporate density (Azure-portal style). Not condescending "for dummies" baby-talk — respect the non-technical user without talking down to them. Stripe, Twilio, and Plaid are cited as developer-experience references (docs clarity, console quality), and Roblox Studio, Scratch, and Canva as onboarding-simplicity references (time-to-first-success, safe places to practice) — both from research this session — but neither is a visual-style reference to copy.

## Design Principles

- Safety-critical state must be legible at a glance, not buried in settings — test vs. live, real money vs. no money, should read from color and copy before anyone reads a word of prose.
- Plain language over technical jargon by default — a control's label should say what happens to a person's business, not what the system calls it internally.
- One primary action per screen; secondary actions stay visually subordinate.
- Never let the UI claim something isn't true yet. (The docs bug this session found — a help page claiming a sandbox existed when it didn't — is the cautionary tale.)
- Extend the existing ink/signal/pulse token system deliberately; don't reach for ad hoc colors outside it.

## Accessibility & Inclusion

WCAG AA baseline (4.5:1 body text contrast minimum). Full keyboard operability — this is a developer tool, and keyboard-first users are common even among the non-technical audience on shared/low-spec devices. Respect prefers-reduced-motion. No color-only meaning, given the audience includes readers who may not distinguish subtle hue differences or may be relying on a translated/non-native-language reading of the UI.
